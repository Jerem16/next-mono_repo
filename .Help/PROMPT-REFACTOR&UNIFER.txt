## 1) Listes — corriger le bug d’index (tri ≠ source)

**But** : garantir que *Modifier/Supprimer* cible toujours le bon item même après tri.

### Option recommandée (robuste) — **API pilotée par ID**

* [ ] **Nouveau contrat `GenericList`** (remplace l’approche “index” par l’ID)
  **Signatures à implémenter** :

  * `getId: (item) => string | number`
  * `onEditById: (id: string | number) => void`
  * `onDeleteById: (id: string | number) => void`
  * (conservent) `onSave`, `onCancel`
  * `editingId: string | number | null` (remplace `editingIndex`)
* [ ] **Implémentation interne** : construire `indexed = items.map((item, originalIndex) => ({ item, originalIndex, id: getId(item) }))`, puis trier **sur `item`** (jamais sur l’index). Les callbacks appellent **toujours** les variantes *ById*.
* [ ] **Managers/pages** :

  * Remplacer l’état `editingIndex` → `editingId`
  * `onEdit`/`onDelete` → `onEditById`/`onDeleteById`
  * Dans les pages (`CreateSection`, `CreateTag`, `CreatePost`, `CreateAuthor`), piloter la sélection par **ID** (et non par position).

### Option alternative (si tu veux rester “index” temporairement)

* [ ] Porter `originalIndex` dans la structure triée et **propager** `originalIndex` dans tous les callbacks.
  *(Moins maintenable ; garde-la seulement en transitoire.)*

**Fichiers à toucher (sans diff)**

* `src/components/Blog/manage/GenericList.tsx` (nouvelle API + utilisation de `getId`)
* `…/authors/AuthorList.tsx`, `…/sections/SectionList.tsx`, `…/posts/PostList.tsx`, `…/tags/TagList.tsx` (passer `getId`, `editingId`, `onEditById`, `onDeleteById`)
* Pages managers (`CreateAuthor.tsx`, `CreateSection.tsx`, `CreatePost.tsx`, `CreateTag.tsx`) : état local `editingId`, handlers par ID

**Critères d’acceptation**

* [ ] Cliquer *Modifier/Supprimer* cible **toujours** le bon item, quel que soit le tri (`byAlpha` ou `byOptionalOrder`).
* [ ] Plus aucune dépendance à l’index trié pour identifier l’élément source.

## 5) Suppression en cascade — politique + services

**Principe** : AppSync/DynamoDB n’ont pas de FK → cascade **applicative**.

* [ ] **Helpers cascade** (`src/entities/core/services/cascade.ts`)

  * `withConcurrency(items, worker, limit)`
  * `deleteEdges(listFn, deleteFn, foreignKey, id)`
  * `setNullBatch(listFn, updateFn, field, id)`
* [ ] **Politique par entité** (`src/entities/models/<entity>/config.ts`)

  * `Post` : `CASCADE` → supprime `PostTag`, `SectionPost` (+ `Comment` si présent)
  * `Tag` : `CASCADE` → supprime `PostTag` (ne touche pas `Post`)
  * `Section` : `CASCADE` → supprime `SectionPost`
  * `Author` : `SET_NULL` → met `Post.authorId = null` (ou `RESTRICT` si tu préfères bloquer)
* [ ] **Étendre les services** : `postService`, `tagService`, `sectionService`, `authorService`

  * Ajouter `deleteCascade({ id })` conformément à la politique
* [ ] **Brancher les pages** : remplacer `service.delete` → `service.deleteCascade`

**Critères**

* [ ] Supprimer un **Tag** → plus aucune arête `PostTag` pour ce tag (posts intacts)
* [ ] Supprimer une **Section** → plus aucune arête `SectionPost` pour cette section
* [ ] Supprimer un **Post** → plus aucune arête `PostTag/SectionPost` liées à ce post
* [ ] Supprimer un **Author** → tous ses posts ont `authorId = null` (ou suppression refusée si `RESTRICT`)
